## II. 核心方案

### 2.1 前台架构

#### A. 主视图：时间长卷（default）
- **结构**：纵向无限滚动，一张一张照片，按拍摄时间从新到旧排列
- **布局**：
  - 手机：单列，左右各 16px 呼吸，照片之间 80-100px 留白
  - 桌面：居中限宽 720px，两侧大量留白
- **照片**：不裁切，保持原始比例，`object-fit: contain`
- **年份标记**：右侧极淡小字（opacity 0.15），12px，像书页页码
- **月份标记**：每张照片上方留白中，极淡日期（opacity 0.15），`2024.09`，同月只显示一次

#### B. 副视图：精选集（降级方案）
- **不单独 tab**，而是时间长卷中的**章节锚点**
- 示例：「第一次一起旅行」「搬进新家」「炉渣来了」「疫情那几年」
- 滚动经过时自然出现，点击可快速导航
- 不硬归类：不是所有照片都必须属于某个精选集
- Admin 管理：创建/编辑/删除/排序精选集

#### C. 封面页
- **内容**：
  - "Our Orbit" 标题（中等字重，手写风格 webfont）
  - 两个小圆点做椭圆轨道运动，周期 20 秒
  - 下方一行小字：**在一起的第 N 天**（实时计算）
- **交互**：不自动跳转，向下箭头暗示，用户主动滑进正文
- **设计**：奶油白背景，极简，没有任何多余元素

#### D. 结尾页
- 最早照片之后，一行手写体文字：
  > 「一切从这里开始。」
- 然后是留白，再是封面轨道动画的微缩版（静止）
- 形成闭环

### 2.2 大图详情页

#### 进入方式
- 点击照片 → 原地放大（Framer Motion `layoutId` shared element transition）
- 背景淡入半透明（不完全黑，用奶油白毛玻璃 `rgba(255, 252, 247, 0.92) + backdrop-filter`）
- URL 变为 `/photo/[id]`（支持直链分享）

#### 布局

**手机端：**
- 照片居中，最大化占屏但保持原比例，上下留 24px+
- 照片下方：
  - 标题（16px，opacity 0.8，不常显）
  - 时间（14px，opacity 0.5，不常显）
  - 若属精选集，显示 pill 标签，可点击跳转该集
- 无按钮，左右滑切换，下滑关闭

**桌面端：**
- 照片最大宽度 70vw，居中偏左
- 右侧留白放标题/时间/标签，纵向居中
- 左右箭头只在 hover 时出现（半透明）
- 键盘 ← → / ESC 支持
- 点背景或 ESC 关闭

#### 信息显示策略
- **第一层**：照片本身（最重要）
- **第二层**：标题（不是每张都有，见下文"标题分层"）
- **第三层**：时间 + 关系锚点
  - 格式：`2024.09` + 如果有意义的话，一句话，如「那年你们在一起第 7 年」或「炉渣来了的第 2 年」
- **第四层**：精选集标签（可点击）

#### 退出时位置保持
- 列表用虚拟滚动（`react-window`），不会卸载
- 记录：scrollOffset + 当前 item index
- 关闭 overlay 时照片缩回原位，列表 scroll 不动
- 直链进入时：先显示详情，关闭后自动 scroll 到该照片在长卷中的位置

### 2.3 标题策略（关键细节）

**核心原则：不是所有照片都值得被命名**

- **80%：无标题**  
  列表中不显示，详情页也留空。这是多数照片的正常状态。
  
- **15%：AI 候选 + 用户一键接受**  
  Admin 上传时点「AI 建议」，系统生成 2-3 个候选标题以 chip 形式出现。
  用户 accept/edit/skip，不改就发布。
  
- **5%：手工精修**  
  真正值得留下一句话的照片（被标星、首页精选、经常查看）。
  这些照片的标题要有温度、有画面感、5-15 个字。

**示例标题：**
- "下雨天的便利店"
- "你说这家店会倒闭的那个下午"
- "第一次去那个海边"
- "阳台上的影子"

**列表中的显示：** 完全不显示标题，页面干净。只有点进详情才看标题。

## III. Admin 后台

### 3.1 单张上传流程

1. 拖照片到上传区 → 立即上传到 Cloudflare R2
2. 自动读 EXIF 提取拍摄时间；无 EXIF 手动选日期
3. 标题输入框（inline editable，可空）
   - 右边「AI 建议」按钮 → 调 API 生成 2-3 候选
   - 候选以 chip 形式出现，点选填入
4. 精选集：多选 dropdown，也可当场输入新建
5. 点「发布」上线，或「存草稿」不公开

### 3.2 批量上传流程（50+ 张）

**Inbox 模式（关键）：**
- 拖入 50 张 → 全部上传，显示缩略图网格
- 批量操作栏：全选/框选 → 批量删除/标记/改精选集
- 时间处理：优先 EXIF，无 EXIF 的高亮提醒

**AI 标题批量生成：**
- 一键为所有无标题照片生成建议
- 逐条显示，用户 accept/edit/skip
- 支持批量接受/拒绝

**最后一键「全部发布」**

### 3.3 精选集管理

- Admin 侧边栏「精选集」tab
- 列表显示所有集合
- 每个集合：改名（inline edit）、改描述、拖拽排序（决定前台显示顺序）、删除（照片不删，只解除关联）
- 集合内照片也可拖拽排序（覆盖默认时间排序）

### 3.4 权限和共用

- 两个人都能上传（via 密码保护的 `/admin`）
- 两个人都能编辑自己的草稿
- 已发布内容都能补充，但**删除/重排序/改日期**需要更高权限或二次确认
- 每张照片显示：uploaded by 谁、last edited by 谁
- 重要字段改动保留简单历史（至少可撤销）

## IV. 技术实现

### 4.1 图片存储

**存储方案：** Cloudflare R2（免费 10GB）
- 原图：R2 存储
- 三个尺寸：thumb（200px）、medium（800px）、full（原图）
- 使用 `next/image` 自动 WebP/AVIF 转换 + CDN

### 4.2 性能策略

**虚拟滚动：** `react-window`
- viewport 外 DOM 不渲染
- 列表中同时只存在视口前后 5 张照片

**加载策略（三段）：**
- 快速滚动：加载 thumb（10-20KB）
- 停下来 300ms：替换为 medium（80-150KB）
- 详情页：先显示 medium 模糊版，full 加载完 crossfade 替换

**弱网优化：**
- 占位用照片的 dominant color 色块（上传时提取存 DB），不用灰色骨架屏
- thumb 极小化
- 使用 CDN + AVIF/WebP 自适应

**500 张不崩：**
- 虚拟滚动 + 懒加载
- 年份分隔线右侧浮动年份指示器（快速滚动时显示）
- 时间长卷可用快速 scroll bar 跳转

### 4.3 技术栈

| 层 | 选择 | 理由 |
|---|---|---|
| 框架 | Next.js App Router | 已有，不换 |
| 样式 | Tailwind CSS + CSS variables | 快、一致 |
| 动画 | Framer Motion | layoutId shared element transition + 滚动动画 |
| 图片 | Next.js `<Image>` + Vercel Image Optimization | 自动优化 |
| 虚拟滚动 | react-window | 500 张不卡 |
| Lightbox | 自写（Framer Motion layoutId） | 不用第三方库，效果更好 |
| 字体 | Inter（正文）+ Caveat（Our Orbit 标题） | 轻量、有辨识度 |
| 存储 | Cloudflare R2 | 免费 10GB + 无限带宽 |
| 部署 | Vercel | 不变 |


## VI. 决策记录

### 为什么不用时间轴组件？
时间轴通常是线性导航，这里照片本身就是线性的（按时间排），所以不需要额外的时间轴 UI。年份标记足够。

### 为什么精选集降级为章节锚点？
- 如果做成独立 view，很容易烂尾（用户手动维护两个分类系统）
- 降级后：系统自动建议聚类，用户只需一键接受/改名，维护成本大幅降低
- 前台体验：滚动过程中自然出现，更顺畅

### 为什么标题分层 80/15/5？
- 200-500 张照片不可能每张都有好标题
- AI 标题生成快但质量参差
- 放任 AI 标题充斥前台会显得廉价
- 分层方式让"好标题"更值钱，也减轻了用户维护负担

### 为什么不用 Three.js？
- 26 张照片已经不值得，500 张更不值得
- 纯 CSS + Framer Motion 足够，手机性能有保障
- 维护成本低

### 为什么用 Cloudflare R2 不用 Vercel Blob？
- R2 免费 10GB 不限带宽，Blob 限带宽
- 200-500 张照片高清版会超 Blob 免费额度
- R2 更稳定

### 为什么 Admin 要有 Inbox 模式？
100+ 张旧照片导入时，"拖进来、一张张编辑标题、发布"流程会直接崩溃。  
Inbox 模式让用户：先全丢进来 → 批量筛 / 删 / 改日期 → AI 批量生成标题 → 逐条 review → 一键发布。  
这是实际可执行的流程。

## VII. 风险与缓解

| 风险 | 缓解方案 |
|---|---|
| 精选集烂尾 | 系统自动建议，用户只需接受/改名 |
| 标题维护崩 | 80% 无标题，标题分层减轻负担 |
| 手机弱网卡顿 | 三层加载策略 + dominant color 占位 |
| 滚动位置丢失 | overlay 方案 + 记录 offset + item index |
| 两人编辑冲突 | 草稿版本保留历史，已发布二次确认 |
| 直链分享隐私 | 直链只显示该张 + 标题，不显示导航 |
